## Цель работы

Ознакомиться с работой сервиса Google Cloud Run: создать и развернуть тестовый сервис, проверить его доступность, изучить логи и метрики, изменить конфигурацию контейнера, а также рассмотреть работу с revisions и распределением трафика между версиями.

## Ход работы

### 1. Создание сервиса Cloud Run

В Google Cloud Console был открыт раздел **Cloud Run** и начато создание нового сервиса. Для лабораторной работы использовался стандартный демонстрационный сервис Hello, предоставляемый Google Cloud. Такой вариант подходит для первичного исследования Cloud Run, так как позволяет быстро развернуть контейнерное приложение без предварительной подготовки собственного Docker-образа.

<img width="1128" height="826" alt="image" src="https://github.com/user-attachments/assets/636e505e-de3a-4913-9ab8-91ccffdee8ef" />

При создании сервиса были выбраны минимальные параметры ресурсов. Минимальная конфигурация позволяет проверить работу Cloud Run без лишнего потребления ресурсов. Также для сервиса был разрешен публичный доступ, чтобы его можно было открыть по URL из браузера.

<img width="1450" height="832" alt="image" src="https://github.com/user-attachments/assets/8f49e56d-d456-4f30-8d35-7d549b91486d" />

После настройки параметров сервис был создан и развернут. Cloud Run автоматически сформировал revision и выдал публичный URL, по которому можно обращаться к приложению.

<img width="1205" height="827" alt="image" src="https://github.com/user-attachments/assets/66ae0d12-effa-4f4e-ae28-6fb26c73be12" />

### 2. Проверка работы сервиса

После завершения деплоя был выполнен переход по URL, предоставленному Cloud Run. Сервис успешно открылся в браузере, что подтверждает корректное развертывание контейнера и доступность приложения извне.

<img width="761" height="726" alt="image" src="https://github.com/user-attachments/assets/d026e21b-5f68-412b-ac6d-79c64419cb7e" />

Таким образом, на первом этапе был создан рабочий Cloud Run service с демонстрационным приложением. Сервис принимал HTTP-запросы и возвращал корректный ответ.

### 3. Анализ логов и метрик

Далее был открыт раздел наблюдаемости сервиса. В Cloud Run доступны встроенные инструменты мониторинга, которые позволяют анализировать обращения к сервису, ошибки, задержки и состояние контейнеров.

<img width="1470" height="848" alt="image" src="https://github.com/user-attachments/assets/937e4c2d-7866-481c-9929-4037e473bd50" />

В разделе логов были отображены события, связанные с запуском сервиса и обработкой HTTP-запросов. После перехода по URL в логах появились записи о входящих запросах. Ошибок при обработке запросов не наблюдалось.

<img width="1468" height="920" alt="image" src="https://github.com/user-attachments/assets/d924cf36-9191-47b0-887b-dd696112d945" />

Метрики показали базовую активность сервиса: количество запросов, задержку обработки и состояние контейнерных экземпляров. Так как сервис тестировался вручную, нагрузка была небольшой. Резких скачков задержки или ошибок не наблюдалось.

<img width="1464" height="834" alt="image" src="https://github.com/user-attachments/assets/a402ad36-574e-4992-a2ed-7212799c1ff2" />

Дополнительно были просмотрены подробные графики метрик. Они подтвердили, что сервис отвечает стабильно, а количество запросов соответствует ручному тестированию через браузер.

<img width="1203" height="443" alt="image" src="https://github.com/user-attachments/assets/f4226263-b491-4849-9537-3a99abec4055" />

### 4. Изменение порта контейнера

Следующим шагом была изменена конфигурация сервиса. В настройках Cloud Run был выбран режим редактирования и деплоя новой revision. Порт контейнера был изменен со стандартного значения `8080` на `8090`.

<img width="480" height="256" alt="image" src="https://github.com/user-attachments/assets/17a81ba4-88b8-4ec0-9bb4-b271a320c599" />

После изменения порта была создана новая revision сервиса. Cloud Run сохраняет предыдущие revisions, поэтому изменение конфигурации не заменяет старую версию напрямую, а создает новую версию, между которыми затем можно распределять трафик.

<img width="1191" height="774" alt="image" src="https://github.com/user-attachments/assets/bde00938-b662-443a-93a4-047b9a7e5cd5" />

После деплоя новой revision сервис был повторно открыт по публичному URL. Сервис продолжил работать корректно. Ошибка при изменении порта не возникла.

<img width="1458" height="869" alt="image" src="https://github.com/user-attachments/assets/5be854a1-577f-445a-9969-8348d5e2694c" />

Такое поведение объясняется тем, что демонстрационный контейнер Cloud Run корректно использует переменную окружения `PORT`. При изменении порта Cloud Run передает новое значение контейнеру, и приложение начинает слушать актуальный порт. Поэтому изменение с `8080` на `8090` не привело к недоступности сервиса.

### 5. Проверка логов и метрик после изменения конфигурации

После изменения порта были повторно проверены логи. В них отображались успешные обращения к сервису, а критических ошибок запуска или обработки запросов не было.

<img width="1202" height="550" alt="image" src="https://github.com/user-attachments/assets/dd8d89ef-f055-4397-a356-32b2f964c1d5" />

Метрики после изменения порта также не показали ухудшения работы сервиса. Количество запросов оставалось небольшим, время отклика было стабильным, ошибок не наблюдалось.

<img width="985" height="486" alt="image" src="https://github.com/user-attachments/assets/40afbd2b-dead-46b0-a2ac-8a9f5f02f527" />

### 6. Работа с revisions и переключением трафика

После изменения конфигурации в Cloud Run появились разные revisions сервиса. Это позволило проверить механизм управления трафиком между версиями. Для Cloud Run такой механизм важен, так как он позволяет постепенно выкатывать новые версии сервиса, сравнивать их поведение и при необходимости быстро откатываться на предыдущую revision.

<img width="618" height="210" alt="image" src="https://github.com/user-attachments/assets/802fdaec-e2e3-4f98-ab29-92b6277ad310" />

Трафик был переключен между версиями сервиса. Так как обе версии использовали один и тот же демонстрационный контейнер и отличались в основном настройкой порта, визуально результат работы сервиса не изменился. При обращении по URL сервис продолжал открываться корректно.

<img width="1200" height="579" alt="image" src="https://github.com/user-attachments/assets/476e59ad-7b04-4d76-8750-54309c712215" />

По результатам проверки можно сделать вывод, что переключение трафика между revisions прошло успешно. Сервис продолжал обрабатывать запросы, а в логах и метриках не было выявлено ошибок, связанных с новой конфигурацией.

### 7. Удаление ресурсов

После завершения проверки созданный Cloud Run service был удален, чтобы не оставлять неиспользуемые ресурсы в Google Cloud.

## Вывод

В ходе лабораторной работы был изучен сервис Google Cloud Run. Был создан и развернут тестовый сервис Hello с минимальными ресурсами, после чего его работа была проверена через публичный URL.

Были рассмотрены встроенные инструменты наблюдаемости Cloud Run: логи и метрики. Логи позволили проверить запуск сервиса и обработку HTTP-запросов, а метрики — оценить количество запросов, задержку и общее состояние сервиса. При тестовой нагрузке сервис работал стабильно, ошибок не возникало.

Также была изменена конфигурация контейнера: порт был изменен с `8080` на `8090`. После деплоя новой revision сервис продолжил работать корректно. Это связано с тем, что демонстрационный контейнер использует переменную окружения `PORT`, которую Cloud Run передает приложению при запуске контейнера.

Дополнительно был изучен механизм revisions и распределения трафика между версиями. Переключение трафика не привело к ошибкам, так как обе версии сервиса были работоспособны. Этот механизм полезен для постепенного внедрения изменений, тестирования новых версий и безопасного отката.

В результате работы были изучены базовые возможности Cloud Run: создание сервиса, деплой контейнера, публичный доступ, мониторинг, изменение конфигурации, работа с revisions и управление трафиком.
