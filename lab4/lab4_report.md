## 1. Описание приложения

В качестве приложения рассматривается **FlowCraft AI** — AI Copilot для генерации workflow для Langflow по текстовому запросу пользователя.

Пользователь описывает аналитическую задачу на естественном языке, например:

> У меня есть датасет по клиентским транзакциям. Нужно проверить гипотезу, что пользователи с высоким средним чеком чаще возвращаются в течение 30 дней. Отфильтруй клиентов по сегментам, посчитай повторные покупки и сравни retention между группами.

После этого система анализирует запрос, выделяет цель анализа, необходимые фильтры, метрики и шаги обработки данных. Затем FlowCraft AI подбирает подходящие компоненты Langflow, собирает workflow для загрузки данных, фильтрации, расчета метрик, вызова LLM для интерпретации результата и экспортирует готовый граф в формате `flow.json`.

---

## 2. Базовая функциональность MVP

MVP-версия приложения включает:

1. ввод текстового запроса через Web UI;
2. определение типа workflow;
3. подбор подходящего шаблона;
4. заполнение параметров компонентов;
5. проверку корректности структуры workflow;
6. генерацию файла `flow.json`;
7. сохранение истории запросов;
8. скачивание готового результата пользователем.

Для MVP не предполагается поддержка всех компонентов Langflow. На первом этапе достаточно ограничить приложение несколькими типовыми сценариями.

| Сценарий | Описание |
|---|---|
| LLM-chain | Простой workflow: пользовательский ввод → prompt → LLM → ответ |
| RAG workflow | Загрузка документа, поиск релевантных фрагментов и генерация ответа |
| Суммаризация | Получение краткого содержания документа или текста |
| Классификация | Отнесение текста к одной из заданных категорий |

Такое ограничение делает MVP реалистичным: система не пытается генерировать любые возможные графы, а работает с понятным набором шаблонов.

## 3. Состояния приложения

Для приложения рассматриваются три состояния:

1. **Этап 1 MVP** — проверка гипотезы с минимальной инфраструктурой.
2. **Этап 2 тестирование** — подключение партнеров и переход к асинхронной обработке.
3. **Этап 3 production** — масштабирование, безопасность, мониторинг и контроль стоимости.

## 4.3. Описание работы системы

1. Пользователь открывает Web UI в браузере.
2. Пользователь вводит запрос на генерацию workflow.
3. Web UI отправляет запрос в Backend API.
4. Backend API работает в Cloud Run.
5. Backend API проверяет корректность запроса.
6. Backend API отправляет промпт в Vertex AI / Gemini.
7. AI-модель возвращает структуру workflow.
8. Backend API определяет тип workflow, выбирает шаблон, заполняет параметры и валидирует результат.
9. Готовый `flow.json` сохраняется в Cloud Storage.
10. Данные о пользователе, запросе и результате сохраняются в Firestore.
11. Пользователь получает ссылку на скачивание `flow.json`.
12. Логи приложения отправляются в Cloud Logging.

## 4.4. Используемые ресурсы

| Ресурс | Назначение |
|---|---|
| Web UI | Интерфейс для ввода запроса и скачивания результата |
| Cloud Run | Запуск Backend API без управления серверами |
| Vertex AI / Gemini | Генерация структуры workflow |
| Cloud Storage | Хранение `flow.json` и связанных файлов |
| Firestore | Хранение пользователей, запросов, истории и метаданных |
| Cloud Logging | Базовое логирование ошибок и времени генерации |

## 4.5. Обоснование выбора ресурсов

**Cloud Run** выбран для MVP, потому что он позволяет запускать backend без ручного управления серверами. Это удобно для начального этапа, когда нагрузка неизвестна и может быть нерегулярной. Cloud Run тарифицируется по фактическому использованию ресурсов после учета free tier, что хорошо подходит для MVP.

**Vertex AI / Gemini** используется как готовая LLM. Для MVP нецелесообразно обучать собственную модель или поднимать GPU-инфраструктуру. Готовый API позволяет быстро проверить идею продукта.

**Firestore** подходит для MVP, так как данные имеют простую структуру: пользователи, запросы, история генераций, статусы и ссылки на файлы.

**Cloud Storage** используется для хранения итоговых `flow.json`, потому что это файловые артефакты. Хранить такие файлы в объектном хранилище логичнее, чем в базе данных.

**Cloud Logging** нужен для базовой отладки: ошибок backend, времени обработки запроса и проблем при генерации.

## 5.2. Схема инфраструктуры

<img width="1683" height="789" alt="stage 2" src="https://github.com/user-attachments/assets/c42eaf22-84fc-4077-b33e-7ad36f9b97f4" />

## 5.3. Описание работы системы

1. Пользователь открывает Web UI.
2. Пользователь отправляет запрос на генерацию workflow.
3. Backend API Service принимает запрос.
4. Backend проверяет аутентификацию, лимиты и корректность промпта.
5. Backend сохраняет запрос и создает задачу в Firestore.
6. Backend публикует задачу в Pub/Sub.
7. Generation Worker получает задачу из очереди.
8. Worker обращается к Component Registry.
9. Component Registry возвращает информацию о компонентах Langflow, их параметрах и правилах совместимости.
10. Worker отправляет подготовленный промпт в Vertex AI / Gemini.
11. AI-модель возвращает структуру workflow.
12. Worker валидирует результат.
13. Worker сохраняет готовый `flow.json` в Cloud Storage.
14. Worker обновляет статус задачи в Firestore.
15. Пользователь получает уведомление о готовности и скачивает `flow.json`.
16. Cloud Logging и Cloud Monitoring собирают логи, метрики и ошибки.

## 5.4. Используемые ресурсы

| Ресурс | Назначение |
|---|---|
| Cloud Run API Service | Прием пользовательских запросов |
| Pub/Sub | Очередь задач на генерацию workflow |
| Cloud Run Generation Worker | Фоновая генерация и валидация workflow |
| Vertex AI / Gemini | Генерация структуры workflow |
| Firestore | Хранение пользователей, запросов, статусов задач и истории |
| Component Registry | Справочник компонентов Langflow |
| Cloud Storage | Хранение `flow.json`, шаблонов и файлов |
| Cloud Logging / Monitoring | Логи, метрики, ошибки и алерты |

## 5.5. Обоснование выбора ресурсов

На втором этапе синхронная обработка становится менее удобной. Генерация workflow может занимать несколько секунд или дольше, особенно если требуется повторная валидация. Поэтому добавляется **Pub/Sub** и отдельный **Generation Worker**.

Разделение API и worker дает несколько преимуществ:

- пользовательский API не блокируется долгой генерацией;
- задачи можно повторять при ошибках;
- worker можно масштабировать отдельно от API;
- система становится устойчивее к пиковым нагрузкам;
- появляется возможность хранить промежуточные статусы задач.

**Component Registry** нужен для того, чтобы AI-модель не генерировала произвольные компоненты. В нем хранятся допустимые компоненты Langflow, параметры и правила совместимости.

## 6.2. Схема инфраструктуры

<img width="1536" height="1024" alt="stage 3" src="https://github.com/user-attachments/assets/f2d52fd6-9f4e-4341-a3d7-1f9244041806" />

## 6.3. Описание работы системы

1. Пользователь или сотрудник партнерской организации открывает Web UI.
2. Запрос проходит через Cloud CDN и HTTPS Load Balancer.
3. API Gateway маршрутизирует запрос к нужной версии API.
4. Backend API Service проверяет аутентификацию и права доступа через Firebase Auth / IAM.
5. Backend проверяет лимиты пользователя и валидирует запрос.
6. Backend сохраняет запрос и создает задачу в базе данных.
7. Backend публикует задачу в Pub/Sub.
8. Generation Worker получает задачу из очереди.
9. Worker проверяет наличие похожего результата в Memorystore Redis.
10. Если подходящий результат есть в кэше, он используется повторно.
11. Если результата нет, Worker получает информацию из Component Registry.
12. Worker формирует промпт и отправляет его в Vertex AI / Gemini.
13. AI-модель возвращает структуру workflow.
14. Worker валидирует результат и собирает итоговый `flow.json`.
15. Worker сохраняет `flow.json` в Cloud Storage.
16. Метаданные результата сохраняются в Cloud SQL / Firestore.
17. Результат может быть добавлен в Memorystore Redis для повторного использования.
18. Backend возвращает пользователю статус и ссылку на скачивание.
19. Cloud Logging, Cloud Monitoring и Error Reporting собирают логи, метрики и ошибки.
20. Secret Manager хранит токены, ключи и конфигурационные секреты.

## 6.4. Используемые ресурсы

| Ресурс | Назначение |
|---|---|
| Cloud CDN | Ускорение отдачи статического контента |
| HTTPS Load Balancer | Единая точка входа и распределение запросов |
| API Gateway | Маршрутизация, лимиты, версии API |
| Firebase Auth / IAM | Аутентификация и управление доступом |
| Cloud Run API Service | Основной backend для пользовательских запросов |
| Pub/Sub | Асинхронная очередь генерации |
| Cloud Run Generation Worker | Генерация, валидация и сохранение workflow |
| Vertex AI / Gemini | LLM для генерации workflow |
| Cloud SQL / Firestore | Основная база данных |
| Cloud Storage | Хранение `flow.json`, шаблонов и файлов |
| Component Registry | Справочник компонентов и правил совместимости |
| Memorystore Redis | Кэширование готовых и похожих результатов |
| Cloud Logging / Monitoring / Error Reporting | Наблюдаемость, алерты и анализ ошибок |
| Secret Manager | Управление секретами |

## 6.5. Обоснование выбора ресурсов

**API Gateway** добавляется на production-этапе, потому что появляется необходимость управлять маршрутизацией, лимитами, версиями API и доступом.

**Cloud CDN и HTTPS Load Balancer** нужны для production-сценария, где появляются пользователи из разных организаций, а приложение должно иметь стабильную точку входа.

**Memorystore Redis** добавляется только на production. Он используется для кэширования похожих результатов и снижения количества дорогих AI-вызовов.

**Cloud SQL / Firestore** используются для хранения пользователей, организаций, ролей, запросов, истории, статусов задач и метаданных workflow. На ранних этапах достаточно Firestore, но на production можно рассмотреть Cloud SQL, если потребуется более строгая реляционная модель данных.

**Cloud Logging, Cloud Monitoring и Error Reporting** нужны для наблюдаемости production-системы: отслеживания ошибок, задержек, количества генераций, отказов worker-сервисов и стоимости эксплуатации.

# 9. Обоснование экономической модели

Основная статья расходов приложения — обращения к AI-модели. Хранение `flow.json`, работа backend и база данных стоят существенно меньше, чем большое количество LLM-вызовов.

Поэтому для production особенно важны:

1. кэширование похожих результатов;
2. ограничение размера промпта;
3. ограничение числа генераций на пользователя;
4. использование шаблонов вместо полной генерации графа с нуля;
5. повторное использование Component Registry;
6. отдельная валидация результата без лишних LLM-вызовов;
7. мониторинг стоимости одного пользователя.

На MVP расходы минимальны, так как Cloud Run, Firestore, Cloud Storage и Cloud Logging при небольшой нагрузке стоят незначительно. На стадии тестирования расходы растут из-за увеличения числа AI-запросов и появления отдельного worker-сервиса. На production основной рост связан с AI-вызовами, кэшированием, мониторингом, балансировкой и базой данных.
